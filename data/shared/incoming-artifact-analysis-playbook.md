# Incoming artifact analysis playbook

This playbook is shared across the Quack.com and Kinitos / NeoEdge archive projects.

Use it whenever new historical material arrives and still needs first-pass analysis.

## Intake goals

- preserve provenance before cleanup
- keep uncertain facts explicit
- separate raw intake from curated historic storage
- classify each item by historical function, not just file extension
- leave enough notes that the next person or next pass can continue without re-discovering everything

## First-pass checklist

For each incoming item, record:

- current filename
- original filename if different or only partially known
- recovered-from location
- file format
- approximate date or date range
- people, product, or company names mentioned
- company-era fit
  - Quack.com
  - Kinitos
  - NeoEdge Networks
  - transition / uncertain
- likely archive lane
  - memories
  - demos
  - artifacts
  - code
- one to three sentence summary
- confidence notes for date, authorship, and attribution

## Lane rules

Use `memories` for:

- recollections
- interview notes
- timeline fragments
- narrative context

Use `demos` for:

- walkthroughs
- screenshots captured for presentation
- slide decks
- audio or video demo material
- runnable demo packages

Use `artifacts` for:

- PDFs
- scans
- diagrams
- archived HTML
- release media
- press or marketing collateral
- binaries when their main value is evidentiary

Use `code` for:

- source trees
- scripts
- build notes
- dependency clues
- technical reconstruction material

## Analysis notes

- Keep the item as found until the first-pass note exists.
- Do not normalize away old names, dates, or version strings if they may matter historically.
- If a document reflects a transition between companies, say that directly.
- If the best date is inferred rather than stated, label it as inferred.
- If a document has multiple plausible homes, choose the primary lane and note secondary relationships.
- If an item appears duplicated, keep the best copy and record where the others were found.

## Suggested processing flow

1. Drop the raw item into the project's intake area.
2. Create a short intake note using the shared template.
3. Decide the historic lane.
4. Move or copy the item into the curated historic location.
5. Add cross-links to related pages, timelines, demos, or code notes when known.

## Shared expectation across projects

Quack and Kinitos should use the same intake language where possible:

- provenance
- date confidence
- company-era fit
- lane classification
- follow-up needed

That consistency makes it much easier to compare eras, merge timelines, and build public exhibit pages later.
