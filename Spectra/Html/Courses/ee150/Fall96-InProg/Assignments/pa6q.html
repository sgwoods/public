As usual, I'm struggling to get going on this problem. Any hints?

In English, the outline of the program is something like:

Read the transformation line
Break it up into two arrays: "starting" and "ending"
While you can successfully read another input line
Generate an output line
Print the output line

How do you read any input line? You use getline.

How do you break the line describing the transformations? You search for a blank and then copy the letters before the blank into the "starting" array and the letters after the blank into the "ending" array.

How do you generate the output line? You go through the input line, determine how each character should be changed, and then put the changed version of that character on the output.

How do you determine how a character should be changed? You look it up in "starting". Its position there is the index in "ending" of the character into which it should be changed. That's the character you then put in the output.

Now I'm struggling to do Part 2. Any hints?

The key to doing Part 2 is figuring out how to process the ranges. Presumably for Part 1, you stored the user's description of how to transform the input into two arrays (the first part in "before", the second part in "after"). For example, if the transformation description was "abcd" "dcba", the "before" string would be "abcd" and the "after" string would be "dcba".

For Part 2, you want to store an expanded version of the user's description in these arrays, where you have expanded each range into the characters that range represents. That is, if the user provides "a-d" and "dcba", you want to create a string "abcd" from the "a-d" string.

By the way, Part 2 is simply an add-on to Part 1, so as usual you should start by copying your solution to Part 1 into a new file.

I don't understand ranges. What's the idea behind them?

Think of the range as a shortcut for specifying everything in between the two letters. If the user says this "a-mn-z n-za-m", it means:

  • Change a-m into n-z, or more specifically, change a into n, b into o, etc..., all the way to m into z, and
  • Change n-z into a-m, or more specifically, change n into a, o into b, etc..., all the way to z into m.
So when someone enters "a-mn-z n-za-m", they are really saying exactly the same thing as
abcdefghijklmnopqrstuvwxyz nopqrstuvwxyzabcdefghijklm
which means change a into n, b into o, and so on, all the way through to z into m.

And now I'm stuck on Part 3. Help?

For part 3, a key task is just reading the input. The input is in three parts. The first is just a number: 

3
You can read this with readNumber, which is just like scanf, except that it's smart enough to read a whole line and doesn't leave carriage returns and other characters hanging around. It works like scanf, in that it expects to be passed a pointer to an integer variable.

The next part of the input is a series of items ending with the ".". 

account-name
full name
id
account-name
full name
id
[a bunch more of these]
.
So you now can use an input-reading loop, but instead of scanf, you should call a function to take care of reading each group. Fortunately, there's a function, readStudentInfo, that will do that for ou. The trick is then to determine what to pass it, but that's just a couple of arrays and their sizes, plus a pointer to a variable in which you can store the id. (In fact, you can pass it the arrays in the structure, and it'll fill those in for you.)

Once you have the table built, you need to handle reading the input after the ., with the ids and the scores. You can do that with with another input-reading-loop, this time using readStudentScores. As your ead each id and associated scores, you process them by calculating the average, taking each id, looking up the student with that id, and storing the average in that record.

That leaves sorting (for which a function is provided) and printing, where you get to write your own function.

In Part 2, what are the restrictions on ranges?

Basically, the only restrictions on ranges are the ones that are explicitly mentioned in the assignment. That is, when a transformation description containing ranges is expanded, they have to "cover" the same number of characters (e.g., an input of "a-c" "m-p" is incorrect, since "a-c" expands into "abc" and "m-p" expands into "mnop") and the range has to go "upwards" (e.g, a range of "a-c" is acceptable, but not "c-a").

Here are a few more input examples you should make sure your program handles. The input "a-wx-z d-za-c" is perfectly fine and is a way to replace each character with the character that is three characters after it in the output. The input is "x-za-w" "a-cd-z" is also fine, since there are no restrictions that the ranges have to be provided in order.

I'm completely confused about the error handling for Part 3. Can you explain it?

The original assignment wanted you to worry about three errors. The first error occurred if there were problems with reading any of the names, account names, or ids. That is, if for some reason you didn't get either a "." or a legal student when you go to read an entry (e.g., there was EOF or there was a problem), then your program was suppsed to write an error message about "Problem with reading student information in the table." You no longer have to worry about writing an error in this case! That is, if there's an error or EOF when you go to read a name, account name, or ID, your program should simply quietly quit. In any case, I'm not planning on providing a test case to check this, so don't worry about it.

The second error is if you can't read the number of assignments. There you still have to write the same error message that appears on the original assignment handout and then your program quits (if your program doesn't know how many assignments there are, it doesn't make sense for it to go on).

The third error is if you read in a an id (along with scores) and you can't find a student name and account name for that id. In that case, you write the "No student with ID" message and keep going. That is, your program does not terminate when it detects this error.

There are plenty of other problems that could occur (such as not being able to read an ID or not being able to read the scores) but you don't have to worry about those, other than your program must terminate quietly if such an error occurs. That is, it is not OK if your program infinite loops or does a "bus error" or something that, but you don't have to have specific error messages.

Do we check for a valid id before or after we read the associated scores?

You should read the id and the scores, and if you are successful in reading them both, then do the lookup on the id. If you don't find the id, then write the error message (and keep going and process any more ids and scores).

How big should I make the array I'm using to hold the input?

Make the array 81 characters in size. The reason for this is that one line on the display is 80 characters (which is as much as we'll let our user's type). But you also need one extra character to hold the null character getline inserts at the end of the string.

How big should I make the arrays I'm using to hold the "before" and "after" characters?

Make both of these 81 as well. If the user doesn't type a blank in the input, then conceivably the "before" could be the full 80 characters. Similarly, if the user types a blank as the first character, then the "after" can be just about as long.

Can we put our functions in front of main rather than after?

The rationale hehind this question is the observation that if a function definition appears before it is called, you don't have to provide a prototype for it in the calling function.

In general, a program is easiest to read if the main program comes first, then the functions it calls, then the functions they call and so on. In addition, explicitly supplying the prototype makes it clear at a glance exactly which functions a particular function is calling.

So although you can put your functions anywhere, it's a good idea to place them after main rather than before.

How do I write a double quote character (")?

Chapter 4 in the text explains how to do this. Basically, you precede the character with a backslash, as in: 

printf("Bad input: \"%s\"\n", line);
Assuming that line contains the string alex, this will write: 
Bad input: "alex"

For Part 2, in what order do I check for errors?

Check for errors in this order:

  • Missing conversion tables
  • Missing blank
  • Dash at start or finish.
  • Back to back dashes.
  • Illegal range.
  • Duplicate letters.
  • Can't do the mapping.

Do I check the errors on both inputs first, or one at a time?

This question refers to the back to back dashes, illegal range, and duplicate letter error messages. Check the first string for back to back dashes, then for an illegal range. Then check the second string for back to back dashes and an illegal range. Then check the first for duplicate letters, then the second for duplicate letters. Finally, check for whether you can do the mapping. (This order is actually the simplest to implement.)

In Part 3, what does the program do right after it writes the message "No student with id"?

It just goes on and processes the next student (presumably without bothing to store this student's scores). This is not a fatal error message.

Are the declarations of countCharacter and findCharacter correct?

Yes and no.

Originally, the helpers.c and helpers.h files defined them as returning a char and taking a character array and a single character declared as an int. The int character parameter is fine--characters are stored as small integers anyways, so there's no problem. However, the char return value limits the program to working correctly for short strings, so it's a problem.

As a result, the two files helpers.c and helpers.h have been changed. If you retrieved them before Sunday evening, April 21st, you should retrieve the updated versions and use them instead.
